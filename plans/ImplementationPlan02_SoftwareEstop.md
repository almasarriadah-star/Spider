# خطة 02 — الإيقاف الطارئ البرمجي (بدون أي تعديل كهربائي)

> الأولوية: 🔴 حرجة وعاجلة. تُصلِح ثغرة وُجدت أثناء التحقق من تنفيذ الخطة 01.
> بديل برمجي بالكامل عن الإيقاف الفيزيائي (الذي نُقل لمجلد «تطويرات وافاق مستقبلية»).

---

## ⚠️ المشكلة العاجلة المكتشفة

أثناء التحقق من تنفيذ الخطة 01 وُجد أن:
- الواجهة (`spider_gait_controller.html`) تستدعي `‎/api/estop` و`‎/api/estop/clear`.
- لكن **هذين المسارين غير موجودين في `web_controller.py`**.
- النتيجة: **زر «إيقاف طارئ» ميت — يرسل طلباً يرجع 404 ولا يفعل شيئاً.**

الخبر الجيد: دوال الإيقاف **منفّذة بالفعل** في `spider/safety.py`:
- `arbiter.emergency_stop()` → تستدعي `power_cut()` (GPIO، تتجاهَل بأمان بلا عتاد) + **`pwm_release_all()`** (تصفّر إشارة PWM لكل القنوات = ترخية برمجية فورية للسيرفوات) + تفرّغ الأهداف.
- `arbiter.clear_estop()` → تلغي الطوارئ وتعيد التشغيل بهدف=الوضع الحالي (لا قفزة).

إذن الإيقاف الطارئ البرمجي **جاهز 90%** — ينقصه فقط ربط مسارين (6 أسطر).

> **لماذا هذا حلّ برمجي بلا كهرباء؟** `pwm_release_all()` تكتب `duty_cycle = 0` لكل قنوات الـ PCA9685. بلا إشارة PWM، سيرفو الـ hobby يتوقف عن توليد عزم (يرتخي). لا حاجة لأي MOSFET أو Relay.

---

## الحل (الكود الكامل)

### 1) أضف المسارين في `web_controller.py`

ضعهما بعد `api_off` (قرب سطر 1030):

```python
# ── الإيقاف الطارئ البرمجي ──
@app.route("/api/estop", methods=["POST"])
def api_estop():
    """إيقاف طارئ فوري: يوقف كل threads الحركة + يصفّر PWM (ترخية السيرفوات)."""
    global gait_running, _force_stop
    _force_stop = True
    gait_running = False
    gait_event.clear()          # أوقف حلقات المشي
    balance.stop()              # أوقف التوازن لو شغّال
    arbiter.emergency_stop()    # يصفّر PWM + يفرّغ الأهداف + يمنع أي كتابة جديدة
    return jsonify({"ok": True, "estop": True})

@app.route("/api/estop/clear", methods=["POST"])
def api_estop_clear():
    """إلغاء الطوارئ: يعيد المحكّم للعمل بهدف=الوضع الحالي (لا حركة مفاجئة)."""
    global _force_stop
    _force_stop = False
    arbiter.clear_estop()
    return jsonify({"ok": True, "estop": False})
```

### 2) تأكد أن `_run_*` تحترم الطوارئ

محرّكات المشي تكتب عبر `smooth_move_leg → set_servos_batch → arbiter.set_target`. وبما أن `emergency_stop()` تضبط `_estop`، فإن `set_target` ترفض أي كتابة جديدة (`if self._estop.is_set(): return False`). إضافةً، الحلقات تفحص `gait_event.is_set()` الذي مُسِح. فالإيقاف مزدوج: المحكّم يرفض، والحلقات تخرج.

> تحسين اختياري: أضف فحص `arbiter.in_estop()` داخل شروط حلقات `while` في دوال المشي ليخرجوا فوراً حتى لو نُسي مسح `gait_event`:
> ```python
> while gait_event.is_set() and not arbiter.in_estop():
> ```

### 3) الواجهة (مضبوطة أصلاً — تحقّق فقط)

`spider_gait_controller.html` يستدعي `‎/api/estop` و`‎/api/estop/clear` وفيه زر «🟢 إلغاء الطوارئ». بعد إضافة المسارين سيعملان فوراً. تأكد أن `emergencyStop()` لا يستدعي `‎/api/calibrate` بعد الآن (أُزيل في الخطة 01 ✅).

## مستويات الإيقاف (للوضوح)

| المستوى | ماذا يفعل | كهرباء؟ |
|---------|-----------|---------|
| **برمجي (الحالي)** | `pwm_release_all()` تصفّر PWM → السيرفوات ترتخي + إيقاف الحلقات + رفض الكتابة | ❌ لا يحتاج |
| فيزيائي (مستقبلي) | `power_cut()` يقطع التغذية عبر GPIO+MOSFET | ✅ يحتاج (مؤجَّل) |

الكود يستدعي الاثنين معاً؛ الفيزيائي يتجاهَل نفسه بأمان لو لا GPIO. فعند تركيب MOSFET لاحقاً يعمل تلقائياً بلا تغيير كود.

---

## اختبار

| اختبار | المتوقّع |
|--------|----------|
| اضغط «إيقاف طارئ» أثناء المشي | المشي يتوقف فوراً، السيرفوات ترتخي (لا عزم)، لا اهتزاز |
| اضغط «إيقاف طارئ» والروبوت واقف | السيرفوات ترتخي |
| حاول تشغيل مشي بعد الطوارئ | يُرفض (المحكّم بحالة estop) |
| اضغط «إلغاء الطوارئ» | يعود الاستعداد، لا حركة مفاجئة |
| شغّل مشي بعد الإلغاء | يعمل طبيعي |

> ⚠️ ملاحظة سلامة: بعد ترخية PWM، الجسم قد **يهبط** لو كان واقفاً (لا عزم يحمله). هذا متوقّع ومقصود في الطوارئ. نفّذ الاختبار والروبوت على سطح منخفض أو ممسوكاً.
